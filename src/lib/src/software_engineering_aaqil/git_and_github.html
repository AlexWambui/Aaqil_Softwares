Introduction

Version Control System - is a software tool that helps record the changes made to the files by keeping track of changes made to the code so a specific version may be called later if needed.

This ensure that all team members are working on the latest version of the file.

Version control is the management of changes to documents, computer programs, large web sites and other collections of information.

Git is a free and open source verson control system.

Git is the tool that enables you pull or push data to the central server.

Github is a git repository (server) hosting service.

It's a company that allows you host a cross platform repository in a remote server.

Github provides:

  • a web-based graphical interface.
  • easy management of code.
  • open source software for version control
  • effective collaboration
  • bug tracker - called issues (great way to keep track of tasks, enhancements and bugs for your project).

A repository is a project or the folder used to organise one single project. It can contain: folders, files, images, videos, spreadsheets.

it is a great open source community which is cloud based tool so the code becomes more visible and anybody can contribute to it now.

So git is the tool that tracks the changes in your code over time while github is a website where you host all your git repositories.

Git commands

Some git commands include:

1. Clone

used to bring a repository that is hosted somewhere like github into a folder on your local machine.

2. add

used to track your files and changes in Git.

to track a specific file, we would use

git add file_name

to track all the files, those created, modified or deleted, we would use a period (.)

git add .

3. Commit

saves your files in Git.

to commit, we use

git commit -m "commit message"

-m represents the message that should have something to do with a what and why you're making that commit.

you can use more that one -m where the first one will represent the title of the commit and the latter the description of that commit.

git commit -m "title of commit" -m "description of the commit."

4. push

uploads git commits to a remote repository like github.

to push, we use

git push origin master

to get avoid having to use origin master each time we want to push, we can set where we want to push by default and this is called an upstream.

git push -u origin master

-u stands for upstream and is the same as using --set-upstream

5. pull

downloads changes from a remote repository to your local machine, the opposite of push.

6. status

shows all the files that have been added, updated or deleted but haven't been saved in a commit yet.

git status

7. remote

to show which remote repository is connected to your local repo, use

git remote -v

to add a remote origin (origin is the short name for the url).

Whenever you use the name origin, it means you're referring to the particular github url like an alias name.

The name of the remote url can also be changed from origin to any other name. Eg. git remote add my_project <origin_url>

command to add a remote origin:

git remote add origin <origin_url>

To rename the remote url, we can use the command:

git remote rename <old-name> <new-name>

To remove the remote, we can use the command:

git remote remove <name>

8. diff

shows the changes made after comparing two versions of code.

git diff branch_name

9. config

To configure you username and user email for git:

user email

git config --global user.email "email_address"

user name

git config --global user.name "user_name"

10. Verification

To avoid having to fill in the username and password each time, generate a Personal Access Token then use thhe command: (although this is considered to be unsafe)

git config --global credential.helper store

Git repository

Creating a new repo

Initialize local directory as a git repo

git init

add files in your local repo to stage them for commit

git add README.md

commit the file to be tracked for changes and prepare the file to be pushed to a remote repo.

git commit -m "first commit"

git branch -M main

add the url for the remote repo where local repo will be pushed

git remote add origin https://github.com/username/repo_name.git

verify the new remote url

git remote -v

push changes in your local repo up to the remote repo

git push -u origin main

to push an existing repo from the command line

git remote add origin https://github.com/username/repo_name.git

git branch -M main

git push -u origin main

shortcut to add, commit and push

git add . && git commit -m "commit_message" && git push

to unstage a file

git reset HEAD your_file

in case of rejection

git push origin main --force

uninitialize a git repo

rm -rf .git

Branching

By default, a git repo has one default branch which can be called master or main though this is just a naming convention.

So branching comes in when we want to add another repo, say for some new feature that we will later merge with the default branch (master or main).

Each branch in this case will have no idea what commits we have made in any other branch. Each branch keeps track of changes in it.

This is useful when we are making changes that might break our code if they are not complete especially when working with many developers. So we can work on the changes in a different branch to ensure the code in the default branch does not break.

To view all your branches, use

git branch

the branch preceded with an asterisk (*) is the one you're currently in.

To switch between branches, use

git checkout branch_name

To create a new branch, add the flag -b

git checkout -b branch_name

after creating the new branch, git automatically switches to the new branch.

To delete a branch locally, use

git branch -D branch_name

Merging

once we have completed the changes we were making on our branch, we can now merge our branch to the default branch using merge.

before merging we might want to see the code we are merging just to double check. We use

git diff branch_name

then to merge

git merge branch_name

README

You can add a README file to a repository to communicate important information about your project.

A README, along with a repository license, citation file, contribution guidelines, and a code of conduct, communicates expectations for your project and helps you manage contributions.

A README is often the first item a visitor will see when visiting your repository.

README files typically include information on:

  • What the project does
  • Why the project is useful
  • How users can get started with the project
  • Where users can get help with your project
  • Who maintains and contributes to the project

If a repository contains more than one README file, then the file shown is chosen from locations in the following order: the .github directory, then the repository's root directory, and finally the docs directory.

If you add a README file to the root of a public repository with the same name as your username, that README will automatically appear on your profile page. You can edit your profile README with GitHub Flavored Markdown to create a personalized section on your profile. For more information, see "Managing your profile README."

A readme file usually has the file extension .md which stands for markdown.

Markdown is simply a way to format your text in files like the readme file.

It has some shortcuts like one # which represent a heading.

Tips

to update a recently pushed commit message, use

git commit --amend

A simpler way to add, commit and push is to concatenate the commands using &&. ie.

git add file_name && git commit -m "commit message" && git push
